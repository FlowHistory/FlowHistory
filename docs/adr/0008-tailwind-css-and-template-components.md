# ADR 0008: Tailwind CSS Migration and Template Componentization

## Status
Implemented

## Context

The Node-RED Backup web UI is built with Bootstrap 5 (vendored, no CDN) and five Django templates: `base.html`, `login.html`, `dashboard.html`, `detail.html`, and `settings.html`. The project has no Node.js toolchain, no build system, and runs as a single Docker container.

Several problems have emerged with the current approach:

1. **Vendored Bootstrap bloat.** The full `bootstrap.min.css` (227 KB) and `bootstrap.bundle.min.js` (79 KB) are shipped even though the app uses a small fraction of Bootstrap's component library (cards, tables, badges, navbar, alerts, form controls, grid). Bootstrap's JavaScript is only used for alert dismissal and the navbar collapse.

2. **Repeated template patterns.** Status/info cards (4-column grids with label + value), card sections with headers, badge rendering, page headers with action buttons, form field groups, and the `restoreBackup()` JavaScript function are duplicated across templates with no reuse mechanism.

3. **Dark mode coupling.** The theme toggle relies on Bootstrap's `data-bs-theme` attribute and `bg-body-tertiary`/`text-body-secondary` semantic classes. Switching CSS frameworks requires rethinking the dark mode implementation.

4. **Custom CSS is minimal.** Only 23 lines of custom CSS exist (`style.css`), covering sticky footer layout, card borders, and table header styling — all trivially expressible in Tailwind utility classes.

Tailwind CSS is a better fit for this project because: (a) it produces only the CSS actually used in templates, resulting in a much smaller payload; (b) utility classes eliminate the need for a separate custom stylesheet; (c) its `dark:` variant provides first-class dark mode support; and (d) the Tailwind standalone CLI binary requires no Node.js, fitting the project's no-npm constraint.

## Decision

### 1. Tailwind CSS Integration via Standalone CLI

Use the Tailwind CSS standalone CLI binary rather than npm. The standalone binary is a single executable that bundles the Tailwind compiler, PostCSS, and autoprefixer with no Node.js dependency.

**Tailwind configuration file** (`tailwind.config.js`) at the project root:

```js
/** @type {import('tailwindcss').Config} */
module.exports = {
  content: [
    "./backup/templates/**/*.html",
    "./backup/forms.py",
  ],
  darkMode: "class",
  theme: {
    extend: {},
  },
  plugins: [],
}
```

Key points:
- `content` scans all Django templates and `forms.py` (which contains widget `attrs` with CSS class names) for class usage.
- `darkMode: "class"` enables toggling via a `dark` class on `<html>`, matching the existing localStorage-based toggle pattern.

**Input CSS file** (`backup/static/backup/css/input.css`):

```css
@tailwind base;
@tailwind components;
@tailwind utilities;
```

**Output CSS file** (`backup/static/backup/css/tailwind.css`) — generated by the CLI, committed to the repository so that `collectstatic` works without the CLI present at runtime.

**Development workflow:** Download the standalone CLI binary for the host platform. Run in watch mode during development:

```bash
./tailwindcss -i backup/static/backup/css/input.css \
              -o backup/static/backup/css/tailwind.css --watch
```

### 2. Dockerfile Build Step

Add a Tailwind build stage to the multi-stage Dockerfile. The standalone CLI is downloaded in a dedicated build stage:

```dockerfile
FROM python:3.13-slim AS tailwind
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends curl && \
    rm -rf /var/lib/apt/lists/*
RUN curl -sL https://github.com/tailwindlabs/tailwindcss/releases/latest/download/tailwindcss-linux-x64 \
    -o /usr/local/bin/tailwindcss && chmod +x /usr/local/bin/tailwindcss
COPY tailwind.config.js ./
COPY backup/templates/ backup/templates/
COPY backup/forms.py backup/forms.py
COPY backup/static/backup/css/input.css backup/static/backup/css/input.css
RUN tailwindcss -i backup/static/backup/css/input.css \
                -o backup/static/backup/css/tailwind.css --minify
```

The generated `tailwind.css` is then copied into the final image before `collectstatic`:

```dockerfile
COPY --from=tailwind /app/backup/static/backup/css/tailwind.css backup/static/backup/css/tailwind.css
```

This keeps the Tailwind CLI out of the final image and ensures the CSS is always rebuilt from the actual template content.

### 3. Template Component Extraction

Create a `backup/templates/backup/components/` directory with reusable partials, invoked via Django's `{% include %}` tag with variables passed through the `with` keyword:

#### 3a. `_stat_card.html` — Status/Info Card

Used in the 4-column status grids on dashboard and detail pages. Replaces the repeated card markup (currently 8 instances across 2 templates).

```django
{% include "backup/components/_stat_card.html" with label="Instance" value=config.name %}
```

Accepts: `label` (string), `value` (string or safe HTML), optional `value_class` for coloring (e.g., `"text-green-500"` for success states).

#### 3b. `_card_section_start.html` / `_card_section_end.html` — Card with Header

The most repeated pattern: a card container with an `<h2>` header and body content. Used in dashboard (backup history), detail (changes summary, tabs, details), and settings (every fieldset group). Since Django `{% include %}` does not support block-level content injection, this component uses a two-part open/close approach:

```django
{% include "backup/components/_card_section_start.html" with title="Backup History" %}
  <!-- card body content here -->
{% include "backup/components/_card_section_end.html" %}
```

#### 3c. `_badge.html` — Badge Rendering

Consolidates badge styling for status badges (success/failed), trigger badges, and node-type badges in the diff view.

```django
{% include "backup/components/_badge.html" with text=backup.get_trigger_display variant="secondary" %}
```

Accepts: `text`, `variant` (success, danger, warning, secondary, info).

#### 3d. `_form_field.html` — Form Field Group

Replaces the repeated label + input + help text + error pattern in `settings.html` (currently ~14 instances).

```django
{% include "backup/components/_form_field.html" with field=form.name %}
```

Renders the label, widget, conditional help text, and conditional error messages with appropriate Tailwind classes. Checkbox fields receive different markup (inline label).

#### 3e. `_alert.html` — Alert/Message Banner

Replaces the Bootstrap alert dismissible pattern in `base.html`.

Accepts: `message`, `tags`.

#### 3f. `_breadcrumb.html` — Breadcrumb Navigation

Extracts the breadcrumb from `detail.html`.

Accepts: `items` (list of `{"label": ..., "url": ...}` dicts, last item has no `url`).

### 4. Shared JavaScript Extraction

#### 4a. `restoreBackup()` function

Currently duplicated verbatim in `dashboard.html` and `detail.html`. Move to `backup/static/backup/js/app.js` as a global function. The CSRF token will be read from a `<meta>` tag added to `base.html`:

```html
<meta name="csrf-token" content="{{ csrf_token }}">
```

The function reads it at call time:

```js
document.querySelector('meta[name="csrf-token"]').content
```

This eliminates the need for inline `{{ csrf_token }}` in template script blocks.

#### 4b. `createBackup()` function

Similarly move from the inline `<script>` in `dashboard.html` to `app.js`.

### 5. Dark Mode with Tailwind

Replace Bootstrap's `data-bs-theme="dark"` mechanism with Tailwind's `class` strategy:

- The `<html>` element gets a `class="dark"` attribute instead of `data-bs-theme="dark"`.
- All dark-mode colors use Tailwind's `dark:` prefix (e.g., `bg-white dark:bg-gray-900`).
- The theme toggle in `app.js` toggles the `dark` class on `document.documentElement` and persists via `localStorage`.
- System color scheme preference detection is added as a fallback:

```js
const saved = localStorage.getItem('theme');
if (saved === 'dark' || (!saved && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
  document.documentElement.classList.add('dark');
}
```

### 6. Bootstrap Removal

After Tailwind migration is complete:

1. Delete `backup/static/backup/vendor/` directory entirely (Bootstrap CSS + JS)
2. Delete `backup/static/backup/css/style.css` (all rules replaced by Tailwind utilities)
3. Remove Bootstrap `<link>` and `<script>` tags from `base.html`
4. Update `forms.py` widget `attrs` to use Tailwind classes instead of Bootstrap classes

Bootstrap's JavaScript was used for exactly two features:
- **Alert dismissal** (`data-bs-dismiss="alert"`): Replace with an `onclick` handler or a few lines in `app.js`.
- **Navbar responsive collapse** (`navbar-expand-md`): The current navbar is simple enough (brand + 3 buttons) that it does not collapse on mobile. If needed later, a small JS toggle suffices.

### 7. Form Widget CSS Classes

Update `NodeRedConfigForm` in `backup/forms.py` to use Tailwind utility classes in widget `attrs`. Define class strings as constants to avoid repetition:

```python
TW_INPUT = "block w-full rounded-md border border-gray-300 px-3 py-2 text-sm shadow-sm focus:border-blue-500 focus:ring-1 focus:ring-blue-500 dark:bg-gray-800 dark:border-gray-600 dark:text-gray-100"
TW_SELECT = TW_INPUT  # same base styles
TW_CHECKBOX = "h-4 w-4 rounded border-gray-300 text-blue-600 focus:ring-blue-500 dark:border-gray-600 dark:bg-gray-800"
```

### 8. Migration Strategy

The migration is executed as a single atomic effort (not incremental) because:
- The project has only 5 templates totaling ~410 lines of HTML
- Running Bootstrap and Tailwind simultaneously doubles payload size and creates class conflicts
- The visual design is simple enough that a full conversion can be done and verified in one pass

**Execution order:**

1. Add `tailwind.config.js` and `input.css`
2. Download the Tailwind standalone CLI for local development
3. Create component partials in `backup/templates/backup/components/`
4. Extract `restoreBackup()` and `createBackup()` into `app.js`; add CSRF meta tag to `base.html`
5. Convert all templates from Bootstrap to Tailwind, replacing repeated patterns with component includes
6. Update `forms.py` widget attrs to Tailwind classes
7. Update `app.js` theme toggle to use `class="dark"`
8. Remove Bootstrap vendored files and `style.css`
9. Update `Dockerfile` with the Tailwind build stage
10. Run `collectstatic`, verify all pages visually in both light and dark modes

## Consequences

**Positive:**
- Significantly smaller CSS payload (estimated ~8–15 KB vs. 227 KB for full Bootstrap)
- No Bootstrap JavaScript dependency (79 KB removed); the two JS features it provided are replaced by ~10 lines of vanilla JS
- Reusable template components reduce duplication and enforce visual consistency
- `restoreBackup()` deduplication eliminates a maintenance hazard
- First-class dark mode with `dark:` variants is more readable than Bootstrap's semantic color system
- System color scheme preference detection added
- Tailwind build in Dockerfile ensures CSS is always in sync with template content
- No Node.js runtime dependency; the standalone CLI is only present during Docker build

**Negative:**
- Tailwind utility classes are verbose in templates (mitigated by component extraction concentrating styling in fewer files)
- Developers must run the Tailwind CLI in watch mode during local development (mitigated by being a single command, and generated CSS can be committed)
- The standalone CLI binary adds to Docker build time on first pull (mitigated by Docker layer caching)
- Class-based dark mode requires explicit `dark:` variants on every color utility (mitigated by small template count and consistent component usage)
- Atomic migration means a larger single PR to review (mitigated by small total template size)
